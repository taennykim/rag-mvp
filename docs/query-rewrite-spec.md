# Query Rewrite Spec

## 1. 목적
- 고객과 상담사의 대화 내역을 바탕으로 상담지식 컨텐츠 검색용 질문을 한 문장으로 만드는 기준을 정리한다.
- backend query rewrite prompt가 참조하는 운영 스펙을 문서로 고정한다.

## 2. 구현 내용
### Context
- 고객이 컨택센터에 전화로 문의중이다.
- 컨택센터 상담사는 답변을 위해 상담지식 컨텐츠에서 필요한 정보를 가져와야 하는데, 이 때 자료를 검색하기 위한 명료한 질문을 한 문장으로 만들고 싶다.

### Role
- 이 Agent 고객과 상담사간의 대화 내역을 읽고 `상담지식 컨텐츠를 검색하기 위한 질문`을 만드는 AI모델이다.

### Instruction
- 아래에 제공된 고객과 상담사간의 대화 내역을 읽고, 상담지식 컨텐츠에서 적절한 문서를 가져와서 답변을 만들기 위한 질문을 한 문장으로 만든다.
- 각 질문은 독립적으로 상담지식에서 내용을 가져오는 데 사용되므로, 고객이 언급한 서비스 혹은 상품이 명시되어야 한다.

### Do
- 상담지식 검색을 위한 질문을 한 문장으로 만든다.
- 상담대화 중에 고객의 마지막 발언 혹은 요청 및 문의 사항이 상담사가 최종적으로 답변해야 하는 내용이므로, 고객의 가장 마지막 발화를 기준으로 검색문을 만든다.
- 다만 고객의 마지막 발화가 지시어, 군더더기 표현, 끊긴 구어체 위주로 모호하면 직전 고객 발화를 함께 참고해 질문 축을 복원한다.

### Don't
- 고객의 의도를 절대 변경하지 말고, 고객의 개인정보는 질문에 포함하지 않는다.
- 상담대화 중간에 상담사가 고객의 문의에 답변을 완료한 질문은 검색문에 포함하지 않는다.
- 강화된 질문 이외의 특수기호, 문구는 절대 기재하지 않는다.
- `그러니까요`, `예를 들면`, `네`, `음`, `그게` 같은 머뭇거림/추임새/대명사 표현을 최종 검색문에 그대로 남기지 않는다.

### Format
- 고객이 설명한 증상이나 문제상황, 고객이 최종적으로 대화의 마지막에서 궁금해했던 내용을 구체적이게 온전한 문장의 검색문의 형태로 만들어준다. 반드시 질문 형태여야 한다.

### 해석 기준
- 마지막 고객 발화만으로 질문 의도가 충분히 드러나면 해당 발화를 중심으로 검색문을 만든다.
- 마지막 고객 발화가 `그런 게 있는지`, `안 되는 경우가 있는지`, `기간이 있는지`처럼 모호하면 직전 고객 발화에서 상품, 계약 유형, 보장 항목, 상황 맥락을 복원한다.
- 검색에 의미 있는 속성은 유지한다.
  - 예: 가입 시점, 상품명, 보험 종류, 계약 형태, 특약명, 서비스명, 업무명
- 검색에 불필요한 군더더기 표현은 제거하되, 질문 의도와 조건은 보존한다.

### 보험 도메인 보강 규칙
- `종신보험`, `평생 보장`, `무조건 다 되는 줄 알았다`, `안 나오는 경우`, `특정한 이유면 안 된다` 같은 표현이 함께 나오면 `사망보장 예외/면책/지급 제한 조건` 질문으로 구체화할 수 있다.
- `2010년 가입`, `표준형 계약`처럼 고객이 직접 언급한 계약 속성은 검색 범위를 좁히는 데 유의미하면 검색문에 유지한다.
- 단, 대화에 없는 보장 항목이나 상품 속성을 임의로 추가하지 않는다.

### 예시 후보
- 대화 요약:
  - 고객이 2010년쯤 가입한 종신보험 표준형 계약을 언급했고, 평생 보장이라고 들었지만 안 나오는 경우가 있는지 문의했다.
- 기대 검색문:
  - `2010년 가입한 종신보험 표준형 계약에서 사망보장을 못받는 때가 있나요?`
