# Retrieval Test Set

## 1. 목적
- 현재 `/chat`의 retrieval 품질을 빠르게 점검할 수 있는 질문 세트를 정의한다.
- 질문별로 어떤 문서가 먼저 검색되어야 하는지 기준을 맞춘다.
- embedding 교체 전후, chunk 전략 조정 전후를 같은 질문 세트로 비교할 수 있게 한다.

## 2. 구현 내용
- 대상 문서:
  - `300233_계약관계자변경.docx`
  - `약관_(무)신한유니버설종신보험_080312.pdf`
  - `산출방법서_신한큐브종합건강상해보험(무배당, 해약환급금 미지급형)_230404_v2.pdf`
- 작성 기준:
  - 기본 파일에서 실제 추출된 텍스트의 초반부를 확인하고 질문 후보를 만들었다.
  - 질문은 현재 retrieval 단계 점검이 목적이므로 정답 문장 전체보다 `맞는 문서 / 맞는 조문 또는 표 / 맞는 핵심 키워드`가 잡히는지에 집중한다.
- 점검 방법:
  - `/chat`에서 질문 입력
  - `Target file = All indexed files` 기준으로 먼저 확인
  - 필요 시 파일별로 다시 좁혀서 확인
  - 기본 pass 기준:
    - top 1 또는 top 3 안에 기대 문서가 포함된다.
    - preview 또는 full chunk에 기대 키워드가 포함된다.
    - 완전히 다른 문서가 top 1을 지속적으로 차지하면 fail로 본다.

## 3. 현재 상태
- 진행중
- retrieval 품질 점검용 1차 질문 세트 초안 작성 완료
- 문서별 질문 축과 기대 source 파일 정의 완료
- 아직 실제 `/chat`에서 질문별 hit 결과 비교 기록은 시작 전

## 4. 이슈 및 문제
- 현재 embedding이 hash 기반이라 의미 유사도보다 표면 키워드에 더 민감할 수 있다.
- `약관`과 `산출방법서`는 둘 다 보험 도메인 용어가 많아 잘못된 문서가 섞여 올라올 수 있다.
- PDF는 `page_number`, `section_header` 추출이 비어 있을 수 있어 source 확인은 우선 `original_name`, `preview`, 핵심 문구 기준으로 본다.
- 이 문서는 retrieval 품질 점검용이며 아직 evaluation dataset의 reference answer 세트는 아니다.

## 5. 다음 작업
- 아래 질문 세트로 `/chat` retrieval 결과를 실제 실행해 hit 순서를 기록한다.
- 질문별 `top_k=3`과 `top_k=5` 결과를 비교한다.
- fail 빈도가 높은 질문은 chunk 전략 또는 embedding 교체 후보로 분류한다.
- 안정화 후 evaluation dataset 후보 질문으로 일부 승격한다.

## 10. 2026-03-26 rerank 보정 결과
- 적용한 보정:
  - Chroma에서 `top_k`보다 넓은 후보를 먼저 조회
  - query token overlap + 문자 n-gram overlap 기준으로 후처리 rerank 추가
- 결과:
  - `일반형은 해지율 항목에서 어떤 상태인가?`는 `fail -> pass`로 개선
  - `이 상품의 적용 이율은 얼마인가?`는 여전히 `fail`
  - `산출방법서에서 m과 t는 무엇을 의미하나?`는 여전히 `fail`
  - `연복리 이율은 얼마인가?`는 여전히 `fail`
- 해석:
  - 간단한 lexical rerank는 표면 키워드가 명확한 질문 1건 정도는 회복시켰다.
  - 하지만 짧은 질의, 수식/기호형 질의, 약관과 산출방법서가 공유하는 용어가 있는 질의는 여전히 전역 검색에서 혼동된다.
  - 따라서 다음 우선순위는 `실제 embedding 모델 교체`가 맞고, 현재 rerank는 보조 수단으로만 유지하는 편이 적절하다.
