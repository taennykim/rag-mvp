## 배치 평가 체계 정리
- 배치 스크립트는 `http://localhost:8000/chat`을 기본 endpoint로 사용하므로, 반드시 RAG 서버에서 backend가 살아 있는 상태에서 실행해야 한다.
- 기본 배치 결과 파일은 `no`, `Question`, `Response`, `Context` 컬럼을 출력하도록 맞췄다.
- 상세 배치 스크립트는 `RewrittenQuery`, `SearchQueries`, `Detail`까지 함께 남겨 query rewrite와 retrieval trace를 한 번에 확인할 수 있게 했다.
- 비교 스크립트는 배치 결과 TSV와 정답 TSV를 받아 answer 일치도와 context 문서 일치도를 함께 판단하도록 구성했다.
- 비교 스크립트 실행 시에는 `python3` 기본 환경이 아니라 backend 가상환경의 Python을 사용하는 것이 안전하다는 점을 확인했다.

## 확인한 현재 동작
- backend가 살아 있으면 `/chat`은 치조골 이식 수술 판례 질의에 대해 `status=200`으로 응답하고, `rewritten_query`, `search_queries`, `retrieved_chunks`를 포함한 JSON을 반환했다.
- 오늘 기준 batch 실패 원인은 answer 품질 문제가 아니라 backend runtime 부재였다.
- 따라서 batch answer 평가를 계속 진행하려면 code 변경보다 RAG 서버 backend runtime 관리가 우선이다.
